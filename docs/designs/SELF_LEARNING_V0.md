## 問題ステートメント

GStack はセッション間で 30 以上のスキルを実行しますが、セッション間では何も学習しません。 A/レビュー
セッションは N+1 クエリ パターンをキャッチし、同じコードベース上の次の /review をキャッチします。
はゼロから始まります。 /ship の実行により、テスト コマンドが検出され、今後の /ship が検出されます。
それを再発見します。 /investigate でトリッキーな競合状態が見つかり、今後のセッションはありません
それについては知っています。

すべての AI コーディング ツールにはこの問題があります。カーソルにはユーザーごとのメモリがあります。クロード・コードは
クロード.md.ウィンドサーフィンには永続的なコンテキストがあります。しかし、それらはどれも複合的ではありません。どれも
学習内容を構造化します。スキルを越えて知識を共有する人はいません。

## 私たちが構築しているもの

セッションやスキル全体で複合化される、プロジェクトごとの組織的な知識。
すべての gstack スキルが読み取ることができる、構造化され、型付けされ、信頼度でスコア付けされた学習
書く。 The goal: after 20 sessions on the same codebase, gstack knows every
アーキテクチャ上の決定、過去のあらゆるバグパターン、そしてそれが間違っていたたびに。

## 北極星

/autoship (リリース 5)。エンジニアリング チーム全員が 1 つのコマンドで対応します。特徴を説明すると、
計画を承認すれば、それ以外はすべて自動的に行われます。 /autoship はこれなしでは機能しません
学習（R1）、レビューの質（R2）、セッションの持続性（R3）、および適応式
(R4)。リリース 1 ～ 4 は、/autoship を実際に動作させるためのインフラストラクチャです。

＃＃ 観客

AI で構築する YC 創設者。実際のコードベースで gstack を 20 回以上実行する人々
1 週間に 1 回実行すると、同じ質問が 2 回行われることに注意してください。

## 状態システム

gstack has four distinct persistence layers. They share storage patterns
(`~/.gstack/projects/$SLUG/` の JSONL) ですが、目的は異なります。

|システム |ファイル |保存するもの | |によって書かれました読む人 |
|----------|------|------|----------|---------|
| **学習** | `learnings.jsonl` |制度上の知識 (落とし穴、パターン、好み) |すべてのスキル |すべてのスキル (前文) |
| **タイムライン** | `timeline.jsonl` |イベント履歴(スキル開始/完了、分岐、結果) |プリアンブル (自動) | /retro、プリアンブル コンテキストの回復 |
| **チェックポイント** | `checkpoints/*.md` |作業状態のスナップショット (決定、残りの作業、ファイル) | /チェックポイント、/船、/調査 |プリアンブル コンテキストの回復、/チェックポイントの再開 |
| **健康** | `health-history.jsonl` |経時的なコード品質スコア (ツールごと、複合) | /健康 | /レトロ、/シップ (ゲート)、/ヘルス (トレンド) |

これらは重複していません。学び＝あなたが知っていること。タイムライン = 何が起こったか。
チェックポイント = あなたがいる場所。健全性 = コードがどれだけ優れているか。それぞれが答える
違う質問。

## リリースロードマップ

### リリース 1: 「GStack Learns」(v0.13-0.14) — 出荷済み

**見出し:** すべてのセッションが次のセッションをより賢くします。

発送されたもの:
- `~/.gstack/projects/{slug}/learnings.jsonl` での学習の持続性
- `/learn` 手動レビュー、検索、プルーニング、エクスポートのスキル
- すべてのレビュー結果の信頼度調整 (表示ルールによる 1 ～ 10 のスコア)
- 観察/推測された学習に対する信頼度の低下 (1 ポイント/30 日)
- プロジェクト間の学習発見 (オプトイン、AskUserQuestion の同意)
- レビューが過去の学習と一致する場合の「学習が適用されました」コールアウト
- /review、/ship、/plan-*、/office-hours、/investigate、/retro への統合

スキーマ:
```json
{
  "ts": "2026-03-28T12:00:00Z",
  "skill": "review",
  "type": "pitfall",
  "key": "n-plus-one-activerecord",
  "insight": "Always check includes() for has_many in list endpoints",
  "confidence": 8,
  "source": "observed",
  "branch": "feature-x",
  "commit": "abc1234",
  "files": ["app/models/user.rb"]
}
```

タイプ: `pattern` | `pitfall` | `preference` | `architecture` | `tool`
出典: `observed` | `user-stated` | `inferred` | `cross-model`

Architecture: append-only JSONL.重複は読み取り時に解決されます (「最新の勝者」)
キー+タイプごと)。 No write-time mutation, no race conditions.

### リリース 2: 「Review Army」 (v0.14.3-0.14.4) — 出荷済み

**見出し:** 各 PR について 10 人の専門審査員が担当します。

発送されたもの:
- 7 つの並列専門サブエージェント: 常時稼働 (テスト、保守性) +
  条件付き (セキュリティ、パフォーマンス、データ移行、API 契約、設計) +
  レッドチーム (大きな差分/重要な発見)
- 信頼スコアを含む JSON 構造化された調査結果 + エージェント間でのフィンガープリントの重複除去
- レビューごとに記録される PR 品質スコア (0 ～ 10) + /レトロ トレンド
- 学習に基づいた専門家のプロンプト、ドメインごとに過去の落とし穴を注入
- 複数の専門家によるコンセンサスが強調され、確認された結果が強化される
- PLAN_COMPLETION_AUDIT による配信の整合性の強化
- チェックリストのリファクタリング: クリティカル カテゴリはメイン パス、スペシャリストに残ります
  レビュー/スペシャリスト/の重点チェックリストに抽出されたカテゴリ

### リリース 2.5: 「陸軍拡張の検討」 — まだ出荷されていません

**見出し:** R2 が安定していることが証明された後に出荷されます。コアループのパフォーマンスをチェックしてください。

事前チェック: R2 品質指標 (PR 品質スコア、スペシャリストのヒット率、
false positive rates, E2E test stability). If core loop has issues, fix those first.

発送対象:
- E1: アダプティブ スペシャリスト ゲーティング、検索実績 0 の自動スキップ スペシャリスト。
  gstack-learnings-log を介してプロジェクトごとのヒット率を保存します。 User can force with --security etc.
- E3: テスト スタブの生成。各スペシャリストは結果とともに TEST_STUB を出力します。
  プロジェクトからフレームワークが検出されました (Jest/Vitest/RSpec/pytest/Go テスト)。
  Fix-First への流れ: AUTO-FIX は修正を適用し、テスト ファイルを作成します。
- E5: クロスレビュー結果の重複除去。以前のレビュー エントリについては gstack-review-read を参照してください。
  以前にユーザーがスキップした検出結果と一致する検出結果を抑制します。
- E7: スペシャリストのパフォーマンス追跡、gstack-review-log 経由でスペシャリストごとのメトリクスをログに記録します。
  タイムラインの統合: スペシャリストの実行は /retro トレンドの timeline.jsonl に表示されます。

### リリース 3: 「セッション インテリジェンス」 (v0.15.0) — 出荷済み

**見出し:** AI セッションは何が起こったかを記憶します。

発送されたもの:
- セッション タイムライン: すべてのスキルの開始/完了イベントが自動的にログに記録されます。
  `~/.gstack/projects/$SLUG/timeline.jsonl`。ローカルのみで、どこにも送信されません。
  テレメトリ設定に関係なく常にオンになります。
- コンテキストの回復: 圧縮またはセッションの開始後、プリアンブルに最近の CEO がリストされます。
  計画、チェックポイント、レビュー。エージェントは最新のコンテキストを読み取り、コンテキストを回復します。
- クロスセッション注入: プリアンブルは、LAST_SESSION と LATEST_CHECKPOINT を出力します。
  現在のブランチ。何かを入力する前に、どこから中断したかが表示されます。
- 予測スキルの提案: 過去 3 回のセッションがパターンに従っているかどうか
  (レビュー、出荷、レビュー)、gstack は次に必要なものを提案します。
- セッション開始時の「おかえり」合成コンテキスト メッセージ。
- `/checkpoint` スキル: 作業状態のスナップショットを保存/再開/一覧表示します。クロスブランチ
  エージェント間の Conductor ワークスペースのハンドオフのリスト。
- `/health` スキル: プロジェクト ツールをラッピングするコード品質スコアキーパー (tsc、バイオーム、
  knip、シェルチェック、テスト）。 0 ～ 10 の複合スコア、傾向追跡、改善
  スコアが下がったときの提案。
- タイムラインバイナリ: `bin/gstack-timeline-log` および `bin/gstack-timeline-read`。
- ルーティング ルール: プリアンブル スキル ルーティングに /checkpoint と /health が追加されました。

デザインドキュメント: `docs/designs/SESSION_INTELLIGENCE.md`

### リリース 4: 「Adaptive Ceremony」 — まだ出荷されていません

**見出し:** GStack は、安全を損なうことなく時間を尊重します。

Ceremony and trust are separate concerns. Ceremony = the set of review/test/QA
steps a PR goes through.信頼 = どのセレモニーを決定するポリシーエンジン
レベルが適用されます。 They interact but don't merge.

発送対象:

**セレモニーレベル:**
- フル: すべての専門家、敵対者、Codex 構造化レビュー、カバレッジ監査、計画
  完成。大きな差分、新機能、移行、認証の変更の場合。
- 標準: 敵対的 + コーデックス、カバレッジ監査、計画完了。中程度の差分については、
  典型的な長編作品。
- FAST: 敵対的のみ。信頼できるプロジェクトに対する、十分にテストされた小さな変更の場合。

**信頼ポリシー エンジン:**
- スコープを認識した信頼。信頼はグローバルではなく、変更クラスごとに獲得されます。履歴のクリーンアップ
  ドキュメントのみの PR は、移行 PR に対する信頼を購入しません。
- 変更クラスの検出: ドキュメント、テスト、構成、フロントエンド、バックエンド、移行、認証、
  インフラ。各クラスには独自の信頼しきい値があります。
- 信頼シグナル: 連続したクリーンなレビュー (クラスごと)、/ヘルス スコアの安定性、
  回帰頻度、テストカバレッジの傾向。
- 信頼は決して急ぐことはありません: 移行、認証/権限の変更、新しい API エンドポイント、
  インフラストラクチャの変化。これらは、信頼レベルに関係なく、常に完全なセレモニーを取得します。
- バイナリ リセットではなく、徐々に劣化します。単一の回帰によってすべての信頼がリセットされるわけではありません。
  その変更クラスの信頼性が 1 レベル低下します。

**範囲の評価:**
- /review、/ship、/autoplan に基づく TINY/SMALL/MEDIUM/LARGE の分類
  diff サイズ、アクセスされたファイル、およびクラスの変更。
- 儀式レベル = f(範囲、信頼、クラス変更)。

**TODO ライフサイクル:**
- 受信した TODO を対話的に承認するための /triage
- /resolve 並列エージェント経由のバッチ解決

### リリース 5: 「/autoship — 1 つのコマンド、フル機能」 — まだ出荷されていません

**見出し:** 機能について説明します。計画を承認します。それ以外はすべて自動です。

/autoship は再開可能なステート マシンであり、線形パイプラインではありません。レビューとQAが可能
作業をビルド/修正に送り返します。圧縮によりどのフェーズも中断される可能性があります。システムは、
順調に回復します。

```
                    ┌──────────┐
                    │  START   │
                    └────┬─────┘
                         │
                    ┌────▼─────┐
                    │ /office- │
                    │  hours   │
                    └────┬─────┘
                         │
                    ┌────▼─────┐
                    │/autoplan │ ◄── single approval gate
                    └────┬─────┘
                         │
              ┌──────────▼──────────┐
              │       BUILD         │ ◄── /checkpoint auto-save
              └──────────┬──────────┘
                         │
              ┌──────────▼──────────┐
              │      /health        │ ◄── quality gate
              │   (score >= 7.0)    │
              └──────────┬──────────┘
                         │ fail → back to BUILD
              ┌──────────▼──────────┐
              │      /review        │
              └──────────┬──────────┘
                         │ ASK items → back to BUILD
              ┌──────────▼──────────┐
              │        /qa          │
              └──────────┬──────────┘
                         │ bugs found → back to BUILD
              ┌──────────▼──────────┐
              │       /ship         │
              └──────────┬──────────┘
                         │
              ┌──────────▼──────────┐
              │ /checkpoint archive │ ◄── preserve, don't destroy
              └─────────────────────┘
```

発送対象:
- /autoship 上記のステート マシンを備えた自律パイプライン。
  各フェーズは timeline.jsonl に書き込みます。チェックポイントは各フェーズの前に自動保存されます。
  圧縮リカバリ: コンテキストリカバリはチェックポイントとタイムラインを読み取り、次の時点で再開します。
  最後に完了したフェーズ。
- 完了時にチェックポイントをアーカイブします (削除ではありません)。回復状態は保存されます
  失敗したオートシップ実行のデバッグ用。
- /ideate ブレインストーミング スキル (並列分岐エージェント + 敵対的フィルタリング)
- /plan-eng-review の調査エージェント (コードベース アナリスト、履歴アナリスト、
  ベストプラクティス研究者、学習研究者）

依存: R1 (研究機関の学習)、R2 (軍隊の品質の確認)、
R3 (永続性のためのセッション インテリジェンス)、R4 (速度のためのアダプティブ セレモニー)。

## リスク登録

### プロキシは精査を省略する許可としてシグナルを送ります
(Codex レビューにより特定、2026 年 4 月 1 日)

/health スコア、クリーンなレビュー履歴、タイムライン パターンは有用なシグナルです。
安全性を証明するものではありません。これらの信号がセレモニーの削減とオートシップを提供する場合、
障害モードはまれで、サイレントで重大度の高いミスです。緩和策:
- 特定の変更クラス (移行、認証、インフラ、新しいエンドポイント) は迅速に処理されません。
- 信頼はバイナリ リセットではなく、徐々に低下します。
- /autoship は、プロジェクトごとの最初の実行時に常に完全なセレモニーを実行します。信頼が得られます。



コンテキストのリカバリにより、間違ったブランチ状態、古いプラン、または無効なプランが挿入される可能性があります
チェックポイント。緩和策:
- チェックポイントには、YAML フロントマターのブランチ名が含まれます。コンテキスト回復フィルター
  現在のブランチによる。
- タイムライン grep は、LAST_SESSION を表示する前にブランチごとにフィルターします。
- 古いアーティファクトの検出: チェックポイントが 7 日以上経過している場合、潜在的なものとしてメモします。
  現在のものとして提示されるのではなく、古いものになります。

### 必要な検証メトリクス
(Codex レビューにより特定、2026 年 4 月 1 日)

R4 (Adaptive Ceremony) を出荷する前に、次の測定を行ってください。
- 予測提案の精度 (ユーザーは提案されたスキルを実行しましたか?)
- 信頼ポリシーのフォールススキップ率 (高速追跡された PR にはマージ後に問題がありましたか?)
- コンテキスト復元の精度 (復元されたコンテキストは実際の状態と一致しましたか?)
- /ヘルススコアと実際のコード品質の相関関係 (高スコアは予測するか)
  本番環境のバグが少ない?)

これらのメトリクスは R3 の使用中に収集され、R4 が出荷される前にレビューされる必要があります。
